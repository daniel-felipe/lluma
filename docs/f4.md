## Epic: F4 · Booking Slot Engine

### Context
F4 is the core scheduling engine that powers the client booking flow (F5). It extends the existing `GetAvailableSlots` action (which already handles schedule/buffer/breaks) to account for real bookings and temporary slot holds, and exposes a public API endpoint that F5 will consume. Without F4, F5 cannot be built — this is the critical-path blocker.

### Assumptions
1. Slot locking is implemented as a `pending` status on the `Appointment` model with a `locked_until` timestamp — no separate `slot_locks` table needed for MVP.
2. `locked_until` is only meaningful when `status = pending`; confirmed/completed/no_show/cancelled appointments are always "real" and don't expire.
3. The public availability endpoint (`GET /barbers/{slug}/slots`) is unauthenticated — no login required to browse availability (required for F5 client flow).
4. Conflict detection considers buffer time on the trailing edge of each existing booking (i.e., a new slot cannot start within `buffer_minutes` after an existing booking ends).
5. The 14-day rolling window is enforced at the API layer (validation), not in `GetAvailableSlots` itself.
6. `AppointmentStatus` enum defines all statuses needed through F10 (pending, confirmed, completed, no_show, cancelled) — avoids schema changes in later features.
7. Expired pending appointments are excluded from conflict detection in-query (not cleaned up proactively at this stage); a scheduled cleanup command is a nice-to-have for DB hygiene only.

### Out of Scope
- Client-facing booking confirmation UI — that is F5.
- Barber daily/weekly agenda UI — that is F6.
- Cancellation/rescheduling — that is F10.
- WhatsApp notifications — that is F8.
- Manual blocks (barber blocking a time range) — deferred to post-MVP.
- Multi-service booking (booking more than one service at a time).

#### [PHASE 2] Conflict Detection

---

**Task 5: Extend `GetAvailableSlots` to filter out booked and locked slots**
- **Type:** Feature
- **Size:** M
- **Priority:** P0 (blocker)
- **Depends on:** Task 2
- **Description:** The existing `GetAvailableSlots` action correctly computes slots from working hours, buffer, and breaks — but ignores existing appointments entirely. Extend it to accept the barber's existing appointments for the date and filter out any slot that overlaps a confirmed booking or an active slot hold. This is the central correctness guarantee of F4.
- **Acceptance Criteria:**
  - [ ] Given a barber with a confirmed appointment at 10:00–10:30 and buffer 0, no slot starting between 10:00 and 10:30 is returned.
  - [ ] Given a barber with buffer 15 min and a confirmed appointment at 10:00–10:30, no slot starting before 10:45 (10:30 + buffer) is returned.
  - [ ] Given a barber with an active pending lock (not expired) at 14:00–14:30, that slot is excluded from results.
  - [ ] Given a barber with an expired pending lock at 14:00–14:30, that slot IS included in results.
  - [ ] Given no appointments, slot output is identical to the current behavior.
- **Tech Notes:** Add a second parameter `array<int, Appointment> $existingAppointments = []` (or a Collection) to `run()`. For each candidate slot, check: no existing appointment satisfies `$appt->starts_at < $slotEnd && $appt->ends_at->modify('+N minutes') > $slotStart` where N = `schedule->buffer_minutes`. Load appointments at the controller/action layer before calling `GetAvailableSlots` to keep the action pure and testable. Use `Carbon` or native `DateTimeImmutable` comparisons consistently.

#### [PHASE 3] Slot Locking

---

**Task 7: Create `HoldSlot` action**
- **Type:** Feature
- **Size:** S
- **Priority:** P1
- **Depends on:** Task 2
- **Description:** When a client starts the booking confirmation step in F5, a temporary `pending` appointment is created to hold the slot for 5 minutes and prevent concurrent bookings. `HoldSlot` encapsulates this logic — it checks that the slot is still free (re-validates availability at hold time), creates the pending record, and returns it. F5 will call this action before showing the confirmation form.
- **Acceptance Criteria:**
  - [ ] Given a free slot, `HoldSlot::run()` creates a `pending` appointment with `locked_until = now() + 5 minutes` and returns it.
  - [ ] Given a slot that is already held by an active pending appointment, `HoldSlot::run()` throws `SlotUnavailableException` (or returns null — decide and document).
  - [ ] Given a slot with only an expired pending appointment, `HoldSlot::run()` succeeds (expired locks are not blocking).
  - [ ] Given a confirmed appointment in the same slot, `HoldSlot::run()` throws `SlotUnavailableException`.
  - [ ] Given two concurrent calls for the same slot, only one succeeds (use `DB::transaction()` with a select-for-update or unique constraint to prevent races).
- **Tech Notes:** Create `app/Actions/HoldSlot.php` via `php artisan make:action "HoldSlot" --no-interaction`. Create `app/Exceptions/SlotUnavailableException.php`. Use `DB::transaction()` wrapping a check-then-insert to minimize race window. The unique constraint approach: add a unique index on `(barber_profile_id, starts_at)` filtered to active statuses — or rely on the transaction + re-check. Keep it simple for MVP: transaction + re-check is sufficient at this scale.

**Task 8: Scheduled command to prune expired slot locks**
- **Type:** Chore
- **Size:** S
- **Priority:** P2
- **Depends on:** Task 2
- **Description:** Expired `pending` appointments (where `locked_until < now()`) accumulate in the database and pollute the appointments table. A scheduled command prunes them daily (or on-demand) to keep the table clean. This does not affect correctness (expired locks are already filtered in queries) but is necessary for DB hygiene in production.
- **Acceptance Criteria:**
  - [ ] Given expired pending appointments in the DB, `php artisan appointments:prune-locks` deletes them.
  - [ ] Given confirmed/completed/no_show/cancelled appointments, the command does not touch them.
  - [ ] Given the command runs, it outputs the count of deleted records.
  - [ ] Given `app/Console/Kernel.php` (or `routes/console.php`), the command is scheduled to run `daily()`.
- **Tech Notes:** Create via `php artisan make:command PruneExpiredSlotLocks --no-interaction`. Delete where `status = 'pending' AND locked_until < now()`. Register in `routes/console.php` using the Laravel 11+ scheduler syntax: `Schedule::command(PruneExpiredSlotLocks::class)->daily()`.
